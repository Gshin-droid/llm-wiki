# Raw-выжимка: Claude Code Agent Teams (официальная документация)

**Источник:** https://code.claude.com/docs/en/agent-teams (актуально на v2.1.178+, прочитано 2026-07-09)
**Метод:** WebFetch полной страницы документации.

## Статус
Экспериментальная фича, выключена по умолчанию. Включение: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` в settings.json (`env`) или окружении. Без переменной команды не создаются и Claude их не предлагает.

## Архитектура
- **Team lead** — основная сессия, спавнит тиммейтов, координирует, синтезирует.
- **Teammates** — отдельные полные инстансы Claude Code, каждый со своим контекстным окном; общаются друг с другом напрямую (в отличие от субагентов).
- **Shared task list** — общий список задач: pending / in progress / completed; поддерживает зависимости (заблокированная задача не берётся, пока зависимость не завершена); claim через file locking (защита от гонок).
- **Mailbox** — система сообщений между агентами; доставка автоматическая, lead не поллит.
- Хранение: `~/.claude/teams/{team-name}/config.json` (удаляется при завершении сессии) и `~/.claude/tasks/{team-name}/` (персистится локально, переживает resume). Имя — `session-` + первые 8 символов session ID. Config руками не редактировать — перезаписывается.
- С v2.1.178: команды создаются неявно при спавне первого тиммейта; `TeamCreate`/`TeamDelete` больше не существуют; `team_name` в Agent tool принимается, но игнорируется.

## Субагенты vs Agent Teams (официальная таблица)
| | Субагенты | Agent Teams |
|---|---|---|
| Контекст | своё окно; результат возвращается вызывающему | своё окно; полностью независимы |
| Коммуникация | только отчёт главному агенту | тиммейты пишут друг другу напрямую |
| Координация | главный агент управляет всем | общий task list, самокоординация |
| Лучше для | сфокусированные задачи, где важен только результат | сложная работа с обсуждением/взаимной критикой |
| Токены | ниже (результаты суммируются в главный контекст) | значительно выше (каждый тиммейт — отдельный инстанс) |

Правило выбора: субагенты — быстрые исполнители, которые отчитываются; teams — когда работникам нужно делиться находками, оспаривать друг друга и координироваться самим.

## Когда teams оправданы (официальные кейсы)
1. Research & review — параллельное исследование разных аспектов + взаимная критика.
2. Новые модули/фичи — каждый владеет своим куском, не мешая другим.
3. Дебаг конкурирующими гипотезами — тиммейты активно опровергают чужие теории ("scientific debate"); побеждающая гипотеза достовернее, потому что параллельное расследование избегает anchoring bias последовательного.
4. Cross-layer координация — frontend/backend/tests у разных владельцев.

НЕ подходят: последовательные задачи, правки одного файла, работа с большим числом зависимостей — там одна сессия или субагенты эффективнее.

## Best practices (официальные)
- Спавн-промпт должен содержать task-specific контекст: тиммейт получает CLAUDE.md/MCP/skills проекта, но НЕ историю разговора lead'а.
- Размер команды: 3-5 тиммейтов для большинства воркфлоу; 5-6 задач на тиммейта; масштабировать только когда работа реально выигрывает от параллельности ("три сфокусированных тиммейта чаще лучше пяти рассеянных").
- Размер задач: самодостаточные юниты с ясным deliverable (функция, тест-файл, ревью). Слишком мелкие — оверхед координации съедает выгоду; слишком крупные — риск потраченных впустую усилий без чекинов.
- Файловые конфликты: два тиммейта в одном файле = перезаписи. Делить работу так, чтобы каждый владел своим набором файлов.
- Новичкам начинать с research/review задач (без записи кода) — ценность параллельности без проблем параллельной имплементации.
- Мониторить и рулить: не оставлять команду без присмотра надолго.
- Plan approval: можно требовать от тиммейта план до имплементации (read-only plan mode, lead одобряет/отклоняет автономно; критерии одобрения задаются в промпте lead'у).
- Роли переиспользовать через subagent definitions (агент-тип как teammate); `tools`-allowlist и `model` уважаются, но `skills`/`mcpServers` frontmatter — нет.
- Хуки качества: TeammateIdle / TaskCreated / TaskCompleted (exit 2 = заблокировать + фидбек).

## Безопасность/permissions
- Тиммейты стартуют с permission-режимом lead'а (включая --dangerously-skip-permissions!). Пер-тиммейт режим — только после спавна.
- SendMessage от другого агента помечается как «от другой Claude-сессии», не от пользователя; тиммейт не может одобрить permission prompt за пользователя; отказ одному тиммейту нельзя обойти через другого; auto-mode классификатор трактует «одобрение», переданное другим агентом, как недоверенный ввод.
- Permission prompts тиммейтов всплывают в сессию lead'а.

## Ограничения (текущие)
- `/resume`//`/rewind` НЕ восстанавливают in-process тиммейтов (lead может писать несуществующим — сказать заспавнить новых).
- Статус задач может отставать (тиммейт забыл отметить completed → зависимые блокируются; чинить вручную/нуджем).
- Одна команда на сессию; нельзя шарить между сессиями.
- Нет вложенных команд: тиммейт не может спавнить своих тиммейтов.
- In-process тиммейт не может запускать фоновых субагентов (только foreground).
- Lead фиксирован — нельзя передать лидерство.
- Split panes: только tmux/iTerm2; не работает в VS Code terminal, Windows Terminal, Ghostty. Дефолт — in-process (работает везде).
- Известные фейлы: lead начинает делать задачи сам вместо ожидания («Wait for your teammates to complete»); lead завершает команду до фактического окончания работ; тиммейты останавливаются на ошибках вместо восстановления.

## Токены
Расход растёт линейно с числом тиммейтов. Для research/review/новых фич — обычно оправдано; для рутины — одна сессия дешевле.
