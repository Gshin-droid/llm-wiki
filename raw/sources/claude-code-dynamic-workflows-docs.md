# Raw: Claude Code — Dynamic Workflows (официальная документация)

Источник: https://code.claude.com/docs/en/workflows ("Orchestrate subagents at scale with dynamic workflows")
Загружено: 2026-07-15 (автономный разведчик, второй прогон за день)

## Ключевые тезисы и разделы (конспект официальной страницы)

**Что это:** A dynamic workflow is a JavaScript script that orchestrates subagents at scale. Claude writes the script for the task you describe, and a runtime executes it in the background while your session stays responsive.

Требует Claude Code v2.1.154+, доступно на всех платных планах + Anthropic API + Bedrock/Google Cloud Agent Platform/Microsoft Foundry. На Pro включается вручную в `/config` (строка Dynamic workflows).

### Когда использовать вместо субагентов/skills/agent teams — таблица сравнения
| | Subagents | Skills | Agent teams | Workflows |
|---|---|---|---|---|
| Что это | Воркер, которого спавнит Claude | Инструкции, которым Claude следует | Lead-агент, супервизирующий peer-сессии | Скрипт, который исполняет runtime |
| Кто решает, что запускать дальше | Claude, по ходу диалога | Claude, следуя промпту | Lead-агент, по ходу | Скрипт |
| Где живут промежуточные результаты | Контекст Claude | Контекст Claude | Общий task list | Переменные скрипта |
| Что переиспользуется | Определение воркера | Инструкции | Определение команды | Сама оркестрация |
| Масштаб | Несколько делегированных задач за ход | Как субагенты | Горстка долгоживущих peers | Десятки-сотни агентов за прогон |
| Прерывание | Рестарт хода | Рестарт хода | Тиммейты продолжают работать | Resumable в той же сессии |

Ключевая идея: workflow переносит план в код. С субагентами/skills/teams Claude — оркестратор по ходу диалога, каждый результат попадает в контекстное окно. Workflow-скрипт сам держит цикл, ветвление и промежуточные результаты — в контекст Claude попадает только финальный ответ. Это также позволяет закодировать повторяемый quality-паттерн (adversarial review находок друг друга, черновик плана с нескольких углов и взвешивание), а не просто больше агентов.

### Встроенный workflow: `/deep-research`
Фанит веб-поиск по вопросу с нескольких углов, фетчит и кросс-чекает источники, голосует по каждому claim, возвращает cited-отчёт с отфильтрованными недостоверными утверждениями. Требует WebSearch tool.

### Как запустить свой workflow
Два пути: (1) явный запрос в промпте — ключевое слово `ultracode` или естественный язык "use a workflow"/"run a workflow" (до v2.1.160 триггером было слово `workflow`); (2) `/effort ultracode` — комбинация xhigh reasoning effort + автоматическая оркестрация workflow для каждой существенной задачи в сессии (требует v2.1.203+, доступно на моделях с поддержкой xhigh).

### Одобрение перед запуском
В CLI показывается список фаз + опции: Yes run it / Yes and don't ask again for this workflow in this project / View raw script / No. Зависит от permission mode: Default/accept edits — спрашивает каждый раз (кроме "не спрашивать снова"); Auto — только при первом запуске, дальше consent запоминается (кроме ultracode — там вообще не спрашивает); Bypass permissions / `claude -p` / Agent SDK — никогда не спрашивает.

Субагенты внутри workflow всегда работают в `acceptEdits` (файловые правки авто-одобрены), но наследуют tool allowlist сессии — команды/веб-фетчи/MCP вне allowlist всё равно могут спросить разрешение по ходу прогона.

### Сохранение workflow как команды
`/workflows` → выбрать прогон → `s` → сохранить в `.claude/workflows/` (проектный, шарится с клонами репо) или `~/.claude/workflows/` (личный, во всех проектах). Работает потом как `/<name>`. При совпадении имён — проектный побеждает личный. Может принимать вход через `args` (глобальная переменная в скрипте).

### Структура скрипта
```javascript
export const meta = {
  name: 'audit-routes',
  description: 'Audit every route handler for missing auth checks',
}

const found = await agent('List every .ts file under src/routes/.', {
  schema: { type: 'object', required: ['files'], properties: { files: { type: 'array', items: { type: 'string' } } } },
})

const audits = await pipeline(found.files, file =>
  agent(`Audit ${file} for missing authentication checks.`, { label: file }),
)

return audits.filter(Boolean)
```
`agent()` спавнит один субагент со своим контекстом; `pipeline()` прогоняет по одному на каждый элемент списка, без барьера между стадиями (в отличие от `parallel()`, который является барьером-синхронизацией). Можно передать `agent()` JSON Schema для структурированного вывода — вынуждает субагента вернуть валидированный объект вместо прозы, ретраится при несовпадении.

### Как исполняется прогон
Runtime исполняет скрипт в изолированном окружении отдельно от диалога; промежуточные результаты — в переменных скрипта, не в контексте Claude. Каждый прогон пишет свой скрипт в файл под директорией сессии (`~/.claude/projects/`) — можно открыть, сравнить с прошлым прогоном, отредактировать и попросить перезапустить с изменённого файла. Runtime отслеживает результат каждого агента по ходу — это делает прогон resumable в той же сессии (агенты, уже завершившиеся, возвращают кэш; недостроенный агент на паузе не сохраняется и стартует заново при resume). При выходе из Claude Code во время прогона следующая сессия стартует workflow заново с нуля.

### Ограничения runtime
- Нет пользовательского ввода посередине прогона (только permission prompts могут поставить на паузу; для sign-off между стадиями — запускать каждую стадию отдельным workflow).
- Нет прямого доступа к файловой системе/shell из самого скрипта — это делают агенты, скрипт только координирует.
- До 16 одновременных агентов (меньше на машинах с ограниченным числом ядер).
- 1000 агентов суммарно за прогон (защита от runaway-циклов).

### Стоимость
Прогон может тратить существенно больше токенов, чем тот же таск в диалоге — считается в лимиты плана как обычная сессия. Рекомендация: сначала прогнать на маленьком куске (одна директория вместо всего репо). Предупреждение "Large workflow" всплывает при >25 запланированных агентов или прогнозе >1.5M токенов (с v2.1.203) — не блокирует прогон, только предупреждает; порог заменяется на agent count из size guideline, если он задан; на сессиях с ultracode предупреждение не показывается (ultracode уже подразумевает согласие на большие прогоны). Каждый агент workflow использует модель текущей сессии, если скрипт явно не роутит стадию на другую модель или не задана переменная окружения `CLAUDE_CODE_SUBAGENT_MODEL`.

### Настройка размера ("Dynamic workflow size" в /config, с v2.1.202)
Значения: `unrestricted` (дефолт, без гайдлайна), `small` (<5 агентов), `medium` (<15), `large` (<50) — это совет модели, не жёсткий лимит: явный запрос другого масштаба в промпте переопределяет гайдлайн. Runtime-каппы (16 concurrent / 1000 total) действуют независимо от этой настройки.

### Отключение
`/config` тумблер, `"disableWorkflows": true` в `~/.claude/settings.json`, переменная окружения `CLAUDE_CODE_DISABLE_WORKFLOWS=1`, организационно — managed settings/admin settings page. При отключении пропадают bundled-команды, `ultracode` больше не триггерит прогон и убирается из меню `/effort`.
