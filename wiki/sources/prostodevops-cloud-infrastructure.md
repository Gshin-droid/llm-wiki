## Ключевые тезисы
- **Определение облака (NIST, 5 признаков)**: самообслуживание по требованию, доступ по сети, пул ресурсов/мультиарендность, эластичность, измеряемая оплата за потребление. "Чужой компьютер" = хостинг, не облако — путаница, которую источник explicitly разбирает.
- **История**: идея — Маккарти, 1961; таймшеринг мейнфреймов 1960-х = прообраз мультиарендности; Salesforce 1999 — первый SaaS; AWS S3+EC2 (2006) — точка отсчёта публичного облака (миф про "лишние серверы после праздничных распродаж" опровергнут — реальная причина в опыте Amazon с обработкой скачкообразного розничного трафика); Docker (2013), Kubernetes + AWS Lambda/serverless (2014).
- **Модели по уровню ответственности провайдера** (снизу вверх): своё железо (bare metal, полный контроль) → **IaaS** (провайдер = железо+виртуализация, пример EC2/Yandex Compute) → **PaaS** (+ ОС и рантайм, пример managed БД) → **FaaS/serverless** (провайдер запускает только функции по событию, оплата за фактическое время выполнения, пример Lambda/Yandex Cloud Functions) → **SaaS** (готовый продукт, пример Gmail).
- **Модели по расположению**: публичное (дёшево, быстро, чужое железо) / приватное (свой ЦОД, для чувствительных данных/по требованию закона) / гибридное (чувствительное — у себя, пики — в публичное; самый частый выбор зрелых компаний) / мультиоблако (несколько провайдеров ради отказоустойчивости и снижения vendor lock-in).
- **Механика под капотом**: гипервизор режет железо на изолированные VM (Xen у раннего EC2 → KVM сегодня, включая почти все российские облака; AWS Nitro выносит виртуализацию на отдельные карты, почти убирая оверхед); контейнеры — изолированные процессы поверх общего ядра (легче и быстрее VM); Kubernetes — оркестрация контейнеров; VPC — программируемая виртуальная сеть; хранилище трёх типов (блочное — быстрый диск для одной машины, объектное/S3-подобное — масштабируемое и дешёвое для бэкапов/медиа, сетевая папка — общий доступ с многих машин); регион = набор ЦОДов, зона доступности = физически отдельный ЦОД со своим питанием для отказоустойчивости.
- **Плюсы облака**: capex→opex, скорость разворачивания, эластичность оплаты по факту нагрузки, managed-сервисы снимают рутину эксплуатации, глобальность разворачивания в несколько кликов.
- **Минусы**: vendor lock-in (тем глубже, чем больше проприетарных сервисов использовано), при стабильно высокой нагрузке 24/7 своё железо может быть дешевле в моменте (обратный тренд возврата на bare metal), исходящий трафик обычно платный и часто недооценённый источник расходов.
- **Специфика РФ**: 152-ФЗ требует хранить/обрабатывать персональные данные россиян физически в РФ с 2015 года + уведомление Роскомнадзора при трансграничной передаче (с 2023); уход AWS/Azure/GCP в 2022 → рост Cloud.ru/Yandex Cloud/Selectel; для критической инфраструктуры и госсектора — сертифицированные отечественные ОС (Astra Linux Special Edition — лидер, ФСТЭК-сертификация; следом RedOS, Alt Linux) вместо Ubuntu/RHEL.
