# Владилен Минин: Vibe coding мёртв — GSD и Superpowers

**Дата загрузки:** 2026-07-10
**Raw:** `raw/web-clipped/Vibe coding мёртв. Что пришло на замену в 2026. GSD & Superpowers/`
**Источник:** видео YouTube, канал [[vladilen-minin]] (14.05.2026)

## О чём источник
Два полных практических кейса (SaaS на DeepSeek API через GSD, контент-генератор через Superpowers) для двух фреймворков агентской оркестрации поверх Claude Code / Codex, которые, по тезису автора, автоматизируют весь цикл разработки (SDLC), а не только написание кода, как вайбкодинг.

## Ключевой тезис: вайбкодинг закрывал 2 фазы SDLC из 6
Классический SDLC: requirements → design → implementation → testing → deployment → maintenance. Вайбкодинг (термин Карпати, см. [[andrej-karpathy]]) автоматизировал только implementation (+ частично testing), оставляя архитектуру/требования человеку. [[gsd-get-shit-done|GSD]] и [[superpowers]] закрывают агентами все шесть фаз — человек становится "оператором-архитектором", который направляет, а не пишет код. Автор явно противопоставляет это как следующую стадию зрелости после [[vibecoding-full-workflow]], а не замену — см. правку в этом концепте.

## GSD (Get Shit Done)
- Позиционируется как "lightweight powerful metaprompting, context engineering and spec-driven development system for Claude Code" — spec-driven: сначала полная спецификация (SPEC.md), затем декомпозиция на фазы.
- Установка: `npx get-shit-done` (аналогично Superpowers).
- Пайплайн: `GSD new project` (по спеке или через wizard) → фаза research (несколько параллельных агентов собирают актуальную документацию, best practices, версии библиотек) → генерация `project.md`, `requirements.md`, `roadmap.md`, `architecture.md`, `features.md`, `pitfalls.md` — детализированная документация, на ручное создание которой у людей ушли бы недели → `GSD discuss phase N` (уточняющие вопросы у человека) → `GSD plan phase N` → `GSD execute phase N` (пишет код, волнами/waves с параллельными агентами) → self-check/verify → человеческая верификация (human verification) перед закрытием фазы.
- **~70 хуков/команд**, из них 6 базовых достаточно для большинства задач; продвинутые — `GSD milestone` (крупная фича), `GSD quick` (быстрый фикс без полного ритуала фаз), `GSD spike` (тест гипотезы), `GSD sketch` (черновые HTML-варианты дизайна), `GSD mapcodebase` (картирование существующего кода через агентов-мапперов), встроенные Security/Code Review/Graph-агенты (обновляют knowledge graph проекта).
- Настройки инициализации: сколько фаз, параллельно ли их исполнять, включать ли git-tracking, делать ли research перед планированием каждой фазы, верифицировать ли план и результат после каждой фазы, какая модель для планирования (в кейсе — Opus для проектирования, Sonnet для реализации: то же разделение, что уже описано в [[claude-code]] раздел "Модели и Effort").
- Позиционируется как более тяжеловесный, токен-затратный, но надёжный вариант — под enterprise-масштаб.

## Superpowers
- Набор скиллов на разные этапы того же цикла (идея → дизайн/спека → план → сборка → проверка → финиш), построен вокруг TDD.
- Ключевой скилл — **`using superpowers`**: аналог диспетчера, который на основе текущего состояния проекта сам решает, какой из остальных скиллов подключить (brainstorming для спеки, writing plans для декомпозиции, TDD для сборки, verification-before-complete для проверки в браузере, finish-development-branch для завершения ветки).
- Устанавливается как plugin (`Plugin → Install → Superpowers`) в Claude Code, Codex CLI, Gemini, OpenCode, Cursor — заявлена универсальность за пределами одного инструмента.
- В демо (кодекс, не Claude Code) `using superpowers` сам предложил brainstorming-этап с визуальным companion-скиллом (страница-макет в браузере для выбора варианта интерфейса до кода), затем writing plans → **subagent-driven execution**: инструмент сам создаёт именованных субагентов под конкретные задачи (worker, code review agent) с явно прописанной ролью и системным промтом, работающих параллельно волнами — практическая реализация паттерна из [[agent-teams]], но легковеснее и без явной команды.
- Автоматически подключает дополнительные обнаруженные на машине скиллы не из своего набора (пример: `frontend-design` для первого экрана, `deploy-to-vercel` в конце) — то есть работает не как жёсткий фреймворк, а как рамочная система с автообнаружением инструментов.
- Артефакты меньше по объёму, чем у GSD (одна spec-страница вместо полного комплекта документов) — позиционируется как более лёгкий вариант под небольшие/некорпоративные проекты.

## Практические наблюдения из кейсов
- **Bypass permissions** ускоряет работу (агент сам без вопросов переходит между этапами discuss→plan→execute), но требует доверия и git как страховки.
- Оба фреймворка расходуют заметно больше токенов, чем одиночный промт (в кейсе — 67% лимита плана Max за фазу GSD длиной ~2 часа) — цена за долгосрочную архитектурную память и меньше хаоса на больших проектах.
- Даже в автоматизированном режиме важно направлять агента (не соглашаться со всеми его рекомендациями по умолчанию — пример: срезание избыточного дробления на фазы).

## Оценка источника
Автор — практикующий разработчик (13+ лет, ex-frontend, ныне консультирует B2B), демонстрирует оба инструмента вживую, без монтажных склеек ("что получилось с первого раза, то и показываю"). Есть реклама собственного платного сообщества/курса, но не аффилирована с GSD/Superpowers как продуктами (оба — независимые open source проекты).

## Проверка безопасности источника
Описание видео и транскрипт не содержат инструкций, адресованных ИИ-агенту, — обычные CTA к зрителю (подписка, сообщество).
