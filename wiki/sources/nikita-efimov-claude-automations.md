## О чём источник
Обзорный тур по четырём способам автоматизации через **Claude Desktop** (не терминал/CLI) — от простого к сложному. Закрывает пробел в вики: до этого источника механика Cowork Scheduler / Routines / `/loop` не была описана отдельно, только упомянута вскользь в [[claude-cowork]].

## Четыре способа (концепт зафиксирован как [[claude-desktop-automation-modes]])
1. **Cowork Scheduler** — задача обычными словами ("каждый день в 21:00 заходи в папку загрузок и сортируй файлы"), работает только пока компьютер включён, требует коннектор **Desktop Commander** (доступ к файлам/папкам/процессам). Примеры: автосохранение фактур из почты для бухгалтерии, автотранскрипция + саммари любимых YouTube-каналов.
2. **Local Routines** (вкладка Code) — по механике то же самое, что Cowork Scheduler, но к сессии можно подключиться из терминала/Claude Code и продолжить работу оттуда.
3. **Cloud Routines** (remote) — выполняются на серверах Anthropic независимо от состояния компьютера пользователя (можно быть в отпуске/выключить ноутбук). Ограничения: минимальный интервал 1 час (для чаще — только local), лимит запусков в сутки по тарифу (Pro — 5, Max — 15, Team — 25), нет доступа к локальным файлам (только к явно подключённым облачным сервисам — почта, календарь, Google Docs, БД). Три типа триггеров, которые можно комбинировать: по расписанию, по вебхуку (входящий запрос от внешнего сервиса), по API — что превращает автоматизацию из таймера в полноценного реактивного агента.
4. **Команда `/loop`** — короткий цикл прямо внутри активной сессии (не новая сессия каждый раз, как у Routines/Scheduler), имеет доступ ко всему контексту текущего разговора; живёт, пока открыта сессия, максимум 7 дней. Личный кейс автора — pomodoro-напоминания через два параллельных `/loop` (30 сек каждые 10 мин, 5 мин каждые 40 мин).

## Технические нюансы
- Каждый запуск Scheduler/Routine — чистая новая сессия (не помнит предыдущий диалог).
- Catch-up: если компьютер был выключен в момент срабатывания, при включении подтягивается только последний пропущенный запуск за 7 дней (не все пропущенные) — важно ставить длительные отлучки на паузу, а не оставлять активным ежечасный мониторинг.
- Обязательная практика: сначала гонять задачу вручную кнопкой Run Now, поправляя инструкцию, и только когда несколько ручных запусков подряд дают нужный результат — ставить на расписание.
- Главный риск автора, вынесенный отдельно: автономность требует чётко написанной инструкции — агент выполнит задачу "по-своему" и может случайно удалить важное, если человек рассчитывает, что агент "сам поймёт" не проговорённый процесс.
