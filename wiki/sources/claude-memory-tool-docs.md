**Загружено:** 2026-07-09
**Raw:** `raw/sources/claude-memory-tool-docs.md`

## Саммари
Официальный, встроенный в Messages API инструмент памяти для агентов на Claude API — закрывает пробел в [[llm-memory-landscape]]: тот обзор описывал Mem0/Zep/Letta как сторонние библиотеки под "память о пользователе/агенте", но не упоминал, что у самой Anthropic есть нативный примитив под ту же задачу. Архитектурно memory tool ближе всего не к Mem0-подобным SaaS-фреймворкам, а к паттерну этой самой вики: директория текстовых файлов, которые агент сам читает/пишет между сессиями — только без человека-куратора.

## Ключевые тезисы
- **Что это:** clients-side инструмент (`{"type": "memory_20250818", "name": "memory"}`) — Claude только запрашивает файловые операции над `/memories`, приложение разработчика их исполняет и возвращает результат. Данные никогда не хранятся на серверах Anthropic (кроме кратковременной обработки запроса).
- **Шесть команд:** `view` (листинг директории/файла, с построчной нумерацией и `view_range`), `create`, `str_replace`, `insert`, `delete`, `rename` — тот же набор примитивов, что текстовый редактор-тул.
- **Доступность:** все модели Claude 4+, generally available без beta-заголовка.
- **Just-in-time context retrieval:** вместо загрузки всего контекста заранее агент читает файлы памяти по мере необходимости — держит активный контекст сфокусированным на текущей задаче.
- **Сочетается с двумя другими механизмами управления контекстом:** context editing (клиент чистит старые tool-результаты) и compaction (сервер сам суммаризирует весь диалог при приближении к лимиту) — для по-настоящему долгих сессий рекомендуют использовать все три вместе: compaction держит контекст маленьким без ручной работы, memory сохраняет то, что должно пережить суммаризацию.
- **Безопасность — целиком ответственность разработчика**, не тула: обязательная защита от path traversal (`/memories/../../secrets.env`), ограничение размера файлов, периодическая чистка неиспользуемой памяти, фильтрация чувствительных данных (Claude обычно сам отказывается писать секреты, но не гарантированно).
- **Multi-session software development pattern** (самый релевантный раздел для вайбкодинга): initializer-сессия заранее готовит progress log + feature checklist + ссылку на init-скрипт; каждая следующая сессия сначала читает эти файлы; сессия обновляет progress log перед завершением; фича считается готовой только после сквозной верификации, а не когда код написан. Официально отсылает к отдельному кейс-стади — см. [[anthropic-long-running-agent-harness]].
