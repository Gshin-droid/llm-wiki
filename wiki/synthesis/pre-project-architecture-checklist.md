# Чек-лист: продумать архитектуру проекта до старта

**Вопрос, на который отвечает эта страница:** что нужно продумать заранее в вайбкодинг-проекте, чтобы не упустить важное и не пришлось переделывать/начинать с нуля.
**Контекст:** у пользователя пока немного базовых знаний архитектуры и безопасности — эта страница призвана закрыть типовые слепые зоны, а не заменить их изучение.
**Собрано из:** [[vibecoding-full-workflow]], [[ai-security-by-design]], [[seven-signs-of-old-junior]], [[10k-website-checklist]].

## 1. Прежде чем писать первый промпт — бриф
Из [[10k-website-checklist]]: чем конкретнее бриф, тем меньше потом придётся переделывать. Обязательно закончить промпт фразой вроде "задай мне уточняющие вопросы" — это момент, где ИИ ловит неоднозначности, которые иначе всплывут через 10 файлов кода. Ответы на эти вопросы определяют весь проект — не торопиться с ними.

## 2. Архитектура на пальцах — проговорить явно
Из [[vibecoding-full-workflow]]: фронтенд (у пользователя, полностью читаем/изменяем им), бэкенд (посредник и фильтр), база данных (общее состояние). Прежде чем генерировать код, попросить ИИ описать implementation plan и вычитать его самому — это точка контроля, а не формальность.

## 3. Проектирование базы данных — частые ошибки новичков
- Каждая запись — уникальный ID
- Добавляются **строки**, а не колонки (не "билет1, билет2" — отдельная таблица билетов со ссылкой на пользователя)
- Изменения схемы — отдельным файлом-миграцией, а не правкой исходной схемы задним числом (иначе агент забывает применить изменения — источник частых "необъяснимых" багов)

## 4. Security by Design — закладывать сразу, не потом
Из [[ai-security-by-design]] — здесь чаще всего теряют новички:
- **Секреты никогда не на фронтенде.** Любой ключ, который стоит денег или даёт доступ (API-ключи генерации, ключи БД) — только в секретах бэкенда. Фронтенд полностью доступен пользователю браузера.
- **Минимальные привилегии.** Не "на всякий случай", а ровно тот минимум прав, который нужен конкретному сервису/пользователю.
- **Безопасный отказ.** При ошибке/сбое система должна становиться безопаснее, а не пропускать запрос дальше "раз сервис лежит".
- **Не логировать и не показывать секреты/стектрейсы пользователю** — это готовая разведка для атакующего.
- **Проверять права при каждом критическом действии**, а не только при входе (токен может быть валиден, но права уже не актуальны).
- **Простота — осознанный выбор**, не следствие незнания. Каждая лишняя зависимость/абстракция — потенциальная дыра и точка отказа.

## 5. Не переусложнять — но и не гнать голую скорость
Из [[seven-signs-of-old-junior]] — два симметричных провала:
- **Переусложнитель**: архитектура "на вырост" под гипотетические сценарии, которых, возможно, никогда не будет. Тест: смог бы объяснить решение стажёру за 5 минут?
- **Скоростной гонщик**: код без тестов/линтеров ради скорости здесь и сейчас. 70-80% времени в реальных проектах уходит на модификацию существующего кода — писать сразу так, чтобы это было возможно дёшево.
- Практический баланс для вайбкодинга: минимальная архитектура, покрывающая **реальный** сценарий использования (не выдуманный), но с секьюрити-принципами из п.4, которые дёшево заложить сразу и дорого добавлять потом.

## 6. Что проверить перед тем как считать MVP готовым
- Батч-правки по ощущению, не по одной точечной правке за раз (дешевле по токенам, целостнее результат) — см. [[10k-website-checklist]]
- Отдельный проход по мобильной версии, если это веб — "адаптивно" ≠ "спроектировано под телефон"
- Секреты не попали в git (проверить `.env`/аналог перед первым push)
- Хотя бы один сценарий "что если сервис недоступен/данные некорректны" — не только happy path

## Как пользоваться
Перед тем как отправлять первый промпт на генерацию нового проекта — пройтись по пп. 1-4 и явно включить релевантные пункты в сам бриф для ИИ (особенно про секреты и минимальные привилегии — сам ИИ-агент их не добавит, если явно не попросить).
