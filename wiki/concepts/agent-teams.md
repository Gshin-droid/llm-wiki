# Agent Teams (команды агентов Claude Code)

## Определение
Оркестрация нескольких **полных** инстансов Claude Code: одна сессия — team lead (координирует, назначает задачи, синтезирует), остальные — teammates, каждый со своим контекстным окном, прямой связью друг с другом (mailbox) и общим task list с зависимостями. Экспериментальная фича (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`), команда создаётся неявно при спавне первого тиммейта.

Источник: [[claude-code-agent-teams-docs]] (официальная документация, прочитана целиком 2026-07-09 — до этого тема была пробелом вики).

## Субагенты vs Teams — правило выбора
**Субагенты** — быстрые исполнители, которые отчитываются вызывающему и не общаются между собой; результат суммируется в контекст главного агента (дёшево). **Teams** — когда работникам нужно делиться находками, оспаривать друг друга и самокоординироваться (дорого: расход токенов линеен по числу тиммейтов).

Эскалация: 1 задача → 1 агент; независимые подзадачи, где важен только результат → субагенты; работа, требующая обсуждения/взаимной критики между работниками → team.

## Когда team оправдана (официальные кейсы)
1. **Research & review** — параллельные расследования разных аспектов + взаимная критика.
2. **Независимые модули/фичи** — каждый тиммейт владеет своим набором файлов.
3. **Дебаг конкурирующими гипотезами** — тиммейты активно опровергают чужие теории; выжившая гипотеза достовернее (параллельность убирает anchoring bias последовательного расследования).
4. **Cross-layer** — frontend/backend/tests у разных владельцев.

**Антипаттерны:** последовательные задачи, правки одного файла, много зависимостей между кусками — одна сессия или субагенты эффективнее.

## Практические правила
- **Размер:** 3-5 тиммейтов, 5-6 задач на каждого; «три сфокусированных лучше пяти рассеянных».
- **Задачи:** самодостаточные юниты с ясным deliverable; слишком мелкие — оверхед съедает выгоду, слишком крупные — риск незамеченной впустую потраченной работы.
- **Файлы:** каждый тиммейт владеет своим набором файлов — два тиммейта в одном файле = перезаписи.
- **Спавн-промпт** обязан нести task-specific контекст: тиммейт получает CLAUDE.md/skills/MCP проекта, но НЕ историю разговора lead'а.
- **Первый опыт** — research/review без записи кода.
- **Не оставлять без присмотра**: мониторить, рулить, синтезировать по ходу.
- **Plan approval** для рискованных задач: тиммейт в read-only plan mode, lead одобряет автономно по критериям из промпта («одобряй только планы с тестами»).
- **Роли** переиспользовать через subagent definitions (тиммейт уважает `tools`/`model` определения, но не `skills`/`mcpServers`).
- **Хуки качества:** TeammateIdle / TaskCreated / TaskCompleted с exit 2 = заблокировать + фидбек (аналог Default-FAIL из [[long-running-agent-harness]] на уровне команды).

## Безопасность
- Тиммейты наследуют permission-режим lead'а в момент спавна — **включая bypass permissions**; менять можно только после спавна.
- Сообщение от другого агента помечается «от другой Claude-сессии», не от пользователя: тиммейт не может одобрить permission prompt за пользователя, отказ одному нельзя обойти пересылкой через другого, «одобрение» из чужих уст — недоверенный ввод.
- Permission prompts тиммейтов всплывают к lead'у — заранее пре-одобрить частые операции, иначе поток прерываний.

## Известные failure modes
- Lead делает задачи сам вместо ожидания тиммейтов → сказать «Wait for your teammates to complete».
- Lead завершает команду до фактического конца работ → сказать продолжать.
- Тиммейт останавливается на ошибке вместо восстановления → дописать инструкции напрямую или заспавнить замену.
- Статус задачи отстаёт (не отмечена completed) → зависимые блокируются, чинить нуджем/вручную.
- `/resume`/`/rewind` не восстанавливают тиммейтов — lead может писать «мёртвым» именам.

## Ограничения
Одна команда на сессию; нет вложенных команд (тиммейт не спавнит тиммейтов); lead фиксирован; in-process тиммейт не запускает фоновых субагентов; split panes только tmux/iTerm2 (не VS Code/Windows Terminal — дефолтный in-process режим работает везде).
