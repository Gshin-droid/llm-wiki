# Dynamic Workflows (скриптовая оркестрация субагентов)

## Определение
Механизм Claude Code (с v2.1.154, доступен на всех платных планах): JavaScript-скрипт, который Claude пишет под задачу, а отдельный runtime исполняет в фоне, пока диалог остаётся отзывчивым. Ключевая идея — **план оркестрации переносится в код**: скрипт сам держит цикл, ветвление и промежуточные результаты (в переменных), а не Claude по ходу диалога — в контекст Claude попадает только финальный результат.

Источник: [[claude-code-dynamic-workflows-docs]] (официальная документация, прочитана целиком 2026-07-15 — до этого тема была пробелом вики).

## Место среди способов запускать многошаговую работу — правило выбора
Официальная документация сравнивает четыре механизма по тому, **кто держит план**:

| | Субагенты | Skills | Agent Teams | Workflows |
|---|---|---|---|---|
| Кто решает, что дальше | Claude, по ходу | Claude, по инструкции | Lead-агент, по ходу | Скрипт |
| Промежуточные результаты | Контекст Claude | Контекст Claude | Общий task list | Переменные скрипта |
| Масштаб | Несколько задач за ход | Как субагенты | Горстка peers | Десятки-сотни агентов |
| Прерывание | Рестарт хода | Рестарт хода | Тиммейты продолжают | Resumable в сессии |

Соотносится с уже описанным в [[agent-teams]] правилом "1 задача → 1 агент; независимые подзадачи → субагенты; нужна дискуссия между работниками → team" — workflow добавляет пятый случай: **когда оркестрацию стоит закодировать как скрипт**, потому что задачи больше, чем одна сессия может держать в голове (аудит всей кодовой базы, миграция 500 файлов), или потому что нужен воспроизводимый quality-паттерн (adversarial review, кросс-чек источников), а не разовый проход.

## Примитивы
- **`agent(prompt, opts)`** — один субагент с изолированным контекстным окном; `opts.schema` (JSON Schema) вынуждает вернуть провалидированный объект вместо прозы, с ретраем при несовпадении — так следующая стадия получает чистые данные, а не текст для парсинга.
- **`pipeline(items, ...stages)`** — прогоняет каждый элемент списка через все стадии независимо, **без барьера** между ними: элемент A может быть на стадии 3, пока B ещё на стадии 1. Дефолт для многостадийной работы — wall-clock равен самой медленной цепочке, а не сумме самых медленных на каждой стадии.
- **`parallel(thunks)`** — барьер: ждёт все задачи разом. Оправдан только когда стадии реально нужны результаты **всех** предыдущих одновременно (дедуп находок перед дорогой проверкой, ранний выход при нулевом счёте, сравнение находок между собой) — не как способ "сначала отфильтровать/смэппить", это делается внутри стадии `pipeline`.
- **`phase(title)` / `log(message)`** — группировка прогресса и текстовые апдейты пользователю.
- **`workflow(nameOrRef, args)`** — запуск другого сохранённого workflow как шага (вложенность только на один уровень).

## Quality-паттерны (то, ради чего это выгоднее "просто больше агентов")
- **Adversarial verify** — N независимых агентов пытаются опровергнуть находку; выживает только если большинство не смогло опровергнуть.
- **Perspective-diverse verify** — те же N агентов, но каждый смотрит через свою линзу (корректность/безопасность/производительность), а не N одинаковых копий.
- **Judge panel** — несколько независимых попыток решения с разных углов, судьи оценивают, синтез забирает лучшее из каждой.
- **Loop-until-dry** — для задач неизвестного размера (поиск багов/edge cases) — крутить раунды поиска, пока K раундов подряд не дадут ничего нового; простой счётчик "пока < N" пропускает длинный хвост.
- **Completeness critic** — финальный агент спрашивает "что упущено?" (немодарный поиск не запущен, находка не проверена) — результат становится следующим раундом работы.

## Запуск
Явный запрос в промпте (ключевое слово `ultracode` или естественная фраза "используй workflow") **или** `/effort ultracode` — режим, при котором Claude сам решает запускать ли workflow для каждой существенной задачи сессии (комбинация xhigh effort + автооркестрация). Удачный прогон можно сохранить как переиспользуемую команду (`.claude/workflows/` проектный, `~/.claude/workflows/` личный) — появляется в `/` автокомплите с `args` на вход.

## Ограничения и стоимость
До 16 одновременных агентов за прогон, 1000 суммарно (защита от runaway-циклов); никакого пользовательского ввода посередине прогона, кроме permission prompts; скрипт не имеет прямого доступа к файловой системе/shell — это делают агенты, скрипт только координирует. Токенов уходит существенно больше, чем в обычном диалоге — рекомендация тестировать на маленьком куске задачи перед полным прогоном; гайдлайн размера в `/config` (`small`/`medium`/`large`) — совет модели, не жёсткий лимит, runtime-каппы действуют независимо.
