# Obsidian-вики для документации кодового проекта (альтернативный паттерн)

## Определение
Из [[bashkardinov-obsidian-claude-guide]]: другое применение связки Obsidian + Claude, чем персональная база знаний ([[persistent-wiki-pattern]]) — вики как живая документация конкретного кодового проекта, для команды разработчиков.

## Структура (отличается от Jarvis-паттерна)
- `daily-changes/` — лог изменений агентом и разработчиком по дням
- `context/` — техзадание, термины, записи встреч, требования
- `structure/` — техническая документация, повторяющая структуру кодовой базы (client/server/database/endpoints); это основной раздел, с которым работает агент, должен полностью отражать текущее состояние кода
- `README.md` — корневой файл со ссылками на все разделы + инструкции для агентов

## Подключение через MCP (альтернатива прямому доступу к файлам)
В отличие от нашего подхода (Claude Code запускается прямо в папке vault, читает файлы напрямую), здесь используется протокол MCP через плагин Obsidian **Local REST API** + сервер (`obsidian-semantic-mcp`, на момент статьи deprecated → `obsidian-mcp-plugin` в бета). Даёт агенту доступ к vault даже когда Obsidian закрыт, но требует больше настройки ( .mcp.json, API-ключи, сертификаты).

## Практические советы источника
- Файл `CLAUDE.md` в корне проекта, можно ссылаться на него явно, если агент "не слушается"
- Загружать контекст небольшими порциями с проверкой на каждом шаге, не доверять модели описать весь проект за один проход
- Следить за размером файлов — просить агента дробить, если раздел разросся
- Подход даёт максимум пользы на крупных/средних проектах; для простого лендинга хватает обычного контекстного окна

## Применимость к рабочим проектам пользователя
Это отдельный, но совместимый с [[vibecoding-full-workflow|вайбкодинг-воркфлоу]] паттерн — можно применять к конкретному `wiki/projects/`-проекту как способ вести живую документацию кода, отдельно от общей персональной вики.
