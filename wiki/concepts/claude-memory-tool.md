# Memory tool (Claude API)

## Определение
Официальный, встроенный в Claude Messages API инструмент, который даёт агенту постоянную память между сессиями — директория файлов (`/memories`), которые Claude сам создаёт, читает, правит и удаляет через шесть команд (`view`, `create`, `str_replace`, `insert`, `delete`, `rename`). Работает **клиент-сайд**: Claude только запрашивает операцию, приложение разработчика её исполняет над своим собственным хранилищем — Anthropic данные не хранит. Источник: [[claude-memory-tool-docs]].

## Зачем нужен: just-in-time context retrieval
Вместо того чтобы тащить всю релевантную информацию в контекст заранее, агент записывает то, что узнал, в файлы и перечитывает их по мере надобности. Это держит активное контекстное окно сфокусированным на текущей задаче — критично для долгих сессий, которые иначе упираются в лимит токенов.

## Место среди других механизмов управления контекстом
- **Context editing** — клиент вычищает старые результаты инструментов.
- **Compaction** — сервер сам суммаризирует весь диалог при приближении к лимиту контекста.
- **Memory tool** — сохраняет то, что должно **пережить** суммаризацию (в отличие от первых двух, это не про уменьшение текущего контекста, а про перенос знания в следующую сессию).
Для долгоживущих агентов Anthropic рекомендует использовать все три вместе.

## Multi-session software development pattern
Практическая рекомендация для проектов, растянутых на много агентных сессий (прямое применение к вайбкодингу):
1. **Initializer-сессия** заранее готовит progress log, feature checklist и ссылку на init/startup-скрипт — до начала содержательной работы.
2. Каждая следующая сессия **сначала читает** эти файлы — восстанавливает состояние без повторного исследования кода.
3. Сессия **обновляет progress log перед завершением**.
4. Фича считается готовой только после **сквозной верификации**, что она реально работает — не когда код написан.

Официальная документация прямо отсылает к детальному кейс-стади этого паттерна на реальном репозитории — см. [[long-running-agent-harness]] (там тот же принцип реализован через git-коммиты + хуки, а не только через memory tool API).

## Безопасность — ответственность разработчика, не тула
Поскольку исполнение файловых операций целиком на стороне приложения:
- **Path traversal** — обязательная защита (`/memories/../../secrets.env` может выйти за пределы директории памяти); валидировать префикс, резолвить путь в каноническую форму, отклонять `../`/URL-encoded варианты.
- **Размер файлов** — ограничивать, давать Claude листать через `view_range` вместо одного гигантского файла.
- **Expiration** — периодически удалять давно неиспользуемую память.
- **Чувствительные данные** — Claude обычно сам отказывается их писать, но это не гарантия — нужна отдельная валидация.

## Архитектурная параллель с этой вики
Это тот же базовый принцип, что и у [[persistent-wiki-pattern]] — директория markdown/текстовых файлов вместо векторной БД или прогона всей истории через контекст каждый раз. Разница — memory tool предназначен для **самостоятельного** ведения агентом своей рабочей памяти внутри одного продукта/сессии без куратора, а паттерн этой вики — человек-куратор явно направляет, что и как записывается, с более строгой структурой (entities/concepts/sources/projects/synthesis) и качественным контролем на входе (Ingest/Lint). Уточняет [[llm-memory-landscape]]: у Anthropic есть нативная альтернатива Mem0/Zep/Letta для тех, кто строит своего агента прямо на Claude API, а не берёт готовый memory-фреймворк.
