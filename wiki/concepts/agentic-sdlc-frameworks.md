# Агентские фреймворки полного SDLC (GSD, Superpowers)

## Определение
Класс фреймворков поверх агентов кодинга (Claude Code, Codex, OpenCode...), которые автоматизируют весь цикл разработки (SDLC: requirements → design → implementation → testing → deployment → maintenance), а не только фазу написания кода, как классический [[vibecoding-full-workflow|вайбкодинг]]. Источник — [[vladilen-minin-gsd-superpowers]], два представителя — [[gsd-get-shit-done]] и [[superpowers]].

## Зачем это нужно (тезис источника)
Вайбкодинг закрывает 2 фазы SDLC из 6 (implementation + частично testing) — работает для прототипов, но на реальных проектах с ростом кодовой базы Claude/агент "теряет контекст" и код превращается в неподдерживаемую кашу. Агентские SDLC-фреймворки решают это архитектурно: явные фазы, спецификации-как-артефакты, долгосрочная память проекта в файлах (а не в контексте одной сессии).

## Общая механика обоих фреймворков
1. **Спецификация первична** — работа начинается с SPEC.md / brainstorming-сессии, а не с промта "напиши мне X".
2. **Декомпозиция на фазы/этапы** — каждая фаза: обсуждение (discuss) → план (plan) → исполнение (execute, часто параллельными "волнами" субагентов) → самопроверка → human verification перед закрытием.
3. **Артефакты как долгосрочная память** — вместо того, чтобы держать весь проект в контексте одной сессии, фреймворк генерирует файлы документации (требования, архитектура, roadmap), к которым новая сессия обращается заново — тот же принцип, что в [[long-running-agent-harness]] (progress-файл вместо памяти модели), только применённый к дизайну, а не только к трекингу прогресса.
4. **Параллельные субагенты под роли** — оба фреймворка порождают именованных субагентов с явной ролью (worker, code reviewer, security agent) для параллельного исполнения — практическая реализация идей [[agent-teams]] без явной команды/оркестратора.

## Различие по весу
| | GSD | Superpowers |
|---|---|---|
| Модель | Spec-driven, ~70 команд/хуков | Скилл-диспетчер (`using superpowers`), TDD-ориентирован |
| Артефакты | Полный комплект документации на фазу | Компактная spec-страница |
| Токены | Высокий расход | Умеренный |
| Масштаб | Enterprise | Небольшие/некорпоративные проекты |
| Автообнаружение сторонних скиллов | Нет | Да (подключает найденные на машине скиллы сам) |

## Место в эволюции практик вики
Позиционируется автором как следующая стадия зрелости после вайбкодинга, не замена — см. пометку в [[vibecoding-full-workflow]]. Пересекается по духу с [[skill-authoring-practical-rules]] (компонуемость вместо монолита) и с [[claude-code]] разделом про модели (Opus для архитектуры/планирования, Sonnet для реализации — то же разделение труда, что оба фреймворка применяют между фазами).
