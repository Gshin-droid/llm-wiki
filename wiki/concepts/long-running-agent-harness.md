# Harness для долгоживущих агентов (multi-session)

## Определение
Официальный паттерн Anthropic для агентов, работающих над задачей через много отдельных сессий (часы/дни), а не в рамках одного диалога — типичный случай вайбкодинга "фича за фичей". Отвечает на проблему: новая сессия начинается без памяти о предыдущей, поэтому согласованный прогресс через границы контекстных окон нужно строить намеренно, а не полагаться на память модели. Источник: [[anthropic-long-running-agent-harness]] (блог + официальный репозиторий `anthropics/cwc-long-running-agents`).

## Три примитива

### 1. Default-FAIL Contract (принуждение к доказательствам)
Каждая фича стартует с `"passes": false` в файле результатов тестов. `PreToolUse`-хук физически не даёт агенту записать успех, пока он не открыл evidence (скриншот, лог, вывод теста) через Read. Не даёт агенту "соврать себе" об успехе — частая проблема автономных длинных прогонов.

### 2. Fresh-Context Evaluator (независимая оценка)
Отдельный субагент **без прав Write/Edit** проверяет diff и скриншоты из контекстного окна, которое никогда не видело саму разработку — оценивает объективно, без "слепоты автора кода". Возвращает `PASS`/`NEEDS_WORK`; находки становятся стартовым промптом следующей сессии-билдера. То же самое, что паттерн evaluator-optimizer из cookbook [[claude-agent-sdk]] (два отдельных вызова `query()` — генератор и оценщик).

### 3. Agent-Maintained Handoff (восстановление состояния через git)
Агент пишет `PROGRESS.md` в начале каждой сессии (что сделано, что дальше), коммитит на чекпоинтах. Backstop-хук на событие `Stop` подхватывает незакоммиченную работу при остановке агента. Следующая сессия сначала перечитывает `PROGRESS.md` — `git log` + структурированный progress-файл заменяют память, которую агент физически не может унести из закрывшейся сессии.

## Быстрый путь: команда `/goal`
Встроенная в [[claude-code]] альтернатива без кастомных хуков и файла-контракта: одна строка с критерием завершения (`/goal every feature in PROGRESS.md is implemented, committed, and its tests pass`), отдельная быстрая модель сама проверяет выполнение после каждого хода. Работает в Claude Code, headless (`claude -p`) и Remote Control.

## Операторские контролы
`kill-switch.sh` (файл `AGENT_STOP` — немедленный останов всех tool calls), `steer.sh` (файл `STEER.md` — перенаправить агента на ходу без рестарта сессии).

## Важная оговорка источника
Репозиторий сам себя описывает как "ships example ingredients, not a turnkey harness" — три примитива независимы друг от друга, не готовое решение "из коробки"; переносятся напрямую в `PreToolUse`/`Stop`-колбэки [[claude-agent-sdk]].

## Связь с этой вики и вайбкодингом
Дополняет (не заменяет) [[vibecoding-full-workflow]]: тот описывает разработку в рамках одной активной сессии с человеком за штурвалом, этот — что делать, когда проект переживает сессию и нужна автономность на несколько итераций подряд. Прямая параллель с практикой ведения реальных рабочих проектов: похожая идея (свой `CLAUDE.md` проекта + git-коммиты как чекпоинты + subagent'ы `security-review`/`verify` как гейты) без формального Default-FAIL Contract и без выделенного fresh-context evaluator-субагента — потенциальное усиление процесса на таких проектах.
