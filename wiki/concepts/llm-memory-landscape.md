# Ландшафт памяти для LLM-агентов (три разные задачи)

## Определение
Из [[shubin-llm-memory-landscape]]: "память для LLM" — обманчиво общий термин. Внутри него три фундаментально разные задачи с разными оптимальными решениями, и это напрямую уточняет наш собственный [[hybrid-memory-architecture|4-элементную гибридную архитектуру]] из более раннего источника.

## Три задачи, три класса инструментов

### 1. Память о пользователе (факты о человеке)
- **Встроенная память чата** (ChatGPT/Claude/Gemini как продукт) — сжатый профиль пользователя в свободной форме, автоматически в системный промпт. Закрывает 80% бытовых случаев, ничего не нужно ставить.
- **[[mem0]]** — то же самое, но как библиотека для своего агента на API
- **[[zep-graphiti]]** — то же плюс temporal queries (когда факт был верен)
- **[[letta]]** — то же плюс полный контроль над policy памяти, управляемый самим агентом

### 2. Память о коде (архитектура репозитория)
- **[[graphify]]** и аналоги — граф вызовов/импортов/наследования через AST-парсинг, не через embedding-поиск (код плохо ложится в семантический поиск: запрос "функции про платежи" тянет все упоминания слова, включая тесты и README)

### 3. Память о знаниях (накопленное исследование/заметки)
- **Obsidian + [[persistent-wiki-pattern|Karpathy LLM Wiki]]** — то, что реализует эта вики. Никакого фреймворка, никакого сервера — просто папка markdown-файлов и хороший CLAUDE.md.

## Обновление (2026-07-09): нативная альтернатива Mem0/Zep/Letta
Источник [[shubin-llm-memory-landscape]] не упоминал, что у самой Anthropic есть официальный примитив под задачу "память для своего агента на API" — **[[claude-memory-tool|Memory tool]]**, встроенный в Messages API инструмент (генерально доступен, без стороннего фреймворка). Архитектурно он ближе не к Mem0/Zep/Letta (SaaS/библиотека с собственной моделью данных), а к тому же принципу, что уже лежит в основе категории 3 выше — директория файлов, которые агент читает/пишет сам между сессиями. Разница с категорией 1 (Mem0 и т.д.): memory tool — низкоуровневый примитив ("голые" файловые операции + системная инструкция), а не готовая память с профилем пользователя из коробки; разработчик сам решает структуру файлов. Для тех, кто строит агента на чистом Claude API, это теперь более прямой путь, чем сторонний фреймворк — стоит рассматривать в первую очередь.

## Важное уточнение к hybrid-memory-architecture
Источник [[mastodont-claude-obsidian-video]] (более ранний, менее технический) называл "Pinecone" единственной альтернативой для масштабирования. Этот источник (более предметный обзор) вообще не упоминает Pinecone в числе основных игроков — вместо этого описывает Mem0/Zep/Letta как более специфичные и актуальные инструменты именно под память **о пользователе/агенте**, а Graphify — под память **о коде**. Vector DB вроде Pinecone — скорее низкоуровневый примитив под капотом (используется, например, в Mem0), а не готовое решение само по себе. См. обновлённую заметку в [[pinecone]].

## Практический вывод для этой вики
Наш случай — чисто категория 3 (память о знаниях). Mem0/Zep/Letta не решают нашу задачу вообще (это инструменты для тех, кто пишет своего агента на API, а не для персональной базы markdown-файлов). Если когда-то понадобится память **о коде** для рабочих вайбкодинг-проектов — кандидат именно Graphify, а не Pinecone/NotebookLM.
