# Вайбкодинг: полный воркфлоу с нуля

## Определение
Сквозной процесс разработки приложения через ИИ-агента на естественном языке, описанный в [[appbusters-vibecoding-45-min]] на примере генератора обложек для YouTube.

## Инструменты
VS Code (Cursor/Antigravity/Codex — по сути тот же VS Code с встроенной подпиской на нейросеть с наценкой; можно всегда переключиться на чистый VS Code + расширение любой модели, чтобы сохранить гибкость) → [[supabase]] (БД, авторизация, хранилище, Edge Functions) → [[openrouter]] (агрегатор моделей для генерации) → GitHub (версионирование) → Vercel/аналогичный хостинг (деплой).

## Ключевые шаги
1. **Терминал** — базовые команды (ls/pwd/mkdir/cd/touch/cat/rm/clear) нужны, потому что агент просит вызывать чувствительные операции (деплой, установка пакетов) вручную — для контроля
2. **Первый промпт** — описать приложение + стек, дать агенту сгенерировать implementation plan до написания кода — это точка контроля, что агент понял задачу верно
3. **Архитектура на пальцах**: фронтенд (у пользователя, можно посмотреть/изменить код), бэкенд (на сервере, фильтрует и защищает данные), база данных (хранит состояние для взаимодействия между пользователями)
4. **Проектирование БД**: у каждой записи уникальный ID; добавляются только строки, а не колонки (не "билет1, билет2", а отдельная таблица билетов со ссылкой на пользователя) — распространённая ошибка новичков
5. **Ревью плана** — прежде чем запускать генерацию, вычитывать предложенную схему БД и убирать самодеятельность агента (лишние поля, не те связи между таблицами)
6. **API** — простыми словами: то же содержимое сайта, но в машиночитаемом формате (JSON); всегда предпочтительно использовать готовые сервисы, а не изобретать своё
7. **Секреты** — ключи, которые могут стоить денег (OpenRouter), никогда не хранятся на фронтенде (он полностью доступен пользователю), только в секретах бэкенда (Supabase Edge Function secrets)
8. **Итеративная разработка** — сделал кусок → протестировал → нашёл баг → описал агенту симптом (не диагноз) → агент чинит → повтор. Миграции БД — каждое изменение схемы отдельным файлом, а не правка изначальной схемы задним числом (иначе агент забывает применить изменения)
9. **Git/GitHub** — commit как сохранение с комментарием, push — заливка в облако; позволяет откатиться, если агент всё сломал
10. **Деплой** — Vercel подхватывает репозиторий с GitHub и переразворачивает сайт при каждом push автоматически

## Наблюдение
Тот же принцип (учебный проект = шаблон для реального коммерческого сервиса) прослеживается и в [[metics-media-10k-website]] — оба источника используют вайбкодинг-демо как прямую основу для потенциального дохода, а не просто обучающий пример.

## Дополнение (2026-07-09): что делать, когда проект переживает одну сессию
Этот воркфлоу описывает разработку в рамках одной активной сессии с человеком за штурвалом. Для случая, когда работа растягивается на много сессий подряд (типичная ситуация на реальном рабочем проекте), Anthropic формализовал отдельный паттерн — см. [[long-running-agent-harness]]: progress-файл + git-коммиты как чекпоинты вместо памяти модели, отдельный агент-оценщик со свежим контекстом, явный контракт "нет доказательства — нет успеха".

## Дополнение (2026-07-10): следующая ступень — агентские SDLC-фреймворки
[[vladilen-minin-gsd-superpowers|Источник]] характеризует описанный здесь вайбкодинг-воркфлоу как автоматизацию только 2 фаз SDLC из 6 (implementation + частично testing) — рабочий подход для прототипов/MVP вроде описанного здесь, но не для растущей кодовой базы, где вручную приходится держать архитектуру, требования и деплой. [[gsd-get-shit-done|GSD]] и [[superpowers]] закрывают агентами все шесть фаз (включая requirements/design/deployment) через spec-driven декомпозицию и параллельных субагентов — не замена этого воркфлоу, а следующая ступень зрелости для проектов, переросших рамки одного MVP. Подробности — [[agentic-sdlc-frameworks]].
