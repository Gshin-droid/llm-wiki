# Практические правила создания Skills вместо промтов

## Определение
Свод практических правил перехода от одноразовых промтов к [[claude-skills|скиллам]] — переносимым, накапливаемым "умениям" Claude. Синтезировано из официальной методологии Anthropic ([[anthropic-official-skills-docs]]), доклада создателей Skills ([[anthropic-code-summit-build-skills-talk]]) и практики сообщества ([[habr-claude-skills-practical-guide]]).

## Когда переходить от промта к скиллу
Эвристика из официальной документации Claude Code: **создавай скилл, когда ловишь себя на вставке одной и той же инструкции/чеклиста/процедуры в чат повторно**, или когда раздел CLAUDE.md разросся в процедуру, а не остался фактом. Ключевое отличие от CLAUDE.md: тело скилла грузится в контекст только при использовании — длинный референсный материал почти ничего не стоит, пока не понадобился, в отличие от CLAUDE.md, который читается целиком каждую сессию.

## Архитектура: progressive disclosure
Три уровня загрузки контекста (не более того, что нужно прямо сейчас):
1. **Система** — только `name`+`description` всех скиллов в system prompt
2. **Активация** — Claude решает, что скилл релевантен → грузит весь `SKILL.md`
3. **Навигация** — доп. файлы (`references/*.md`) грузятся по требованию через Bash, а не сразу

Это делает объём контекста, который можно "упаковать" в скилл, практически неограниченным — файловая система и code execution заменяют удержание всего в промте разом.

## Три слоя SKILL.md
| Слой | Вопрос | Содержание |
|---|---|---|
| **Description** | Когда брать скилл? | Триггер в frontmatter — единственный механизм, по которому Claude решает подключать скилл |
| **Instructions** | Как выполнить задачу? | Плейбук, пошаговая процедура |
| **Tools** | Чем выполнить? | Скрипты/API-обёртки — детерминированный код вместо AI-вычислений: дешевле, быстрее, повторяемо |

Большинство ограничиваются первыми двумя слоями и считают, что готово — реальная сила в третьем (`scripts/`), но именно туда доходят не все.

## Правило: Description — это всё
Description — единственное, что видит Claude при выборе скилла из списка; тело файла на решение не влияет.
- Объём: ~100 слов в метаданных
- Явное "when to use", а не просто резюме содержимого — это условие срабатывания, а не аннотация
- "Pushy" тон: явно перечислить кейсы, а не быть обтекаемым (расплывчатое описание не подцепится)
- Разные формулировки одного и того же намерения пользователя
- Плохо: `"Build dashboards"` → Хорошо: `"How to build a simple fast dashboard... use whenever user mentions dashboards, data visualization, internal metrics — even if they don't explicitly ask for a 'dashboard'"`
- Метод проверки: 20 eval-запросов (should-trigger / should-not-trigger), с упором на **near-misses** — запросы, которые звучат похоже, но требуют другого скилла

## Структура файлов
```
skill-name/
├── SKILL.md          # frontmatter (name, description) + тело < 500 строк
├── scripts/           # детерминированный код, повторяющаяся логика
├── references/        # документация >300 строк или domain-specific, грузится по требованию
└── assets/            # шаблоны, иконки, статичные файлы
```
- **Правило 500 строк**: тело SKILL.md, приближающееся к пределу — сигнал выносить контент в `references/` с явными указателями (`See references/config-guide.md`)
- **Разделяй взаимоисключающие контексты** по разным файлам (напр. `aws.md`/`gcp.md`/`azure.md`) — экономит токены, Claude грузит только релевантный
- **Сигнал для scripts/**: если в нескольких test-кейсах Claude заново пишет один и тот же код (`build_chart.py`, `create_pdf.py`) — вынести в bundled script один раз

## Композиционность, а не монолит
Один гигантский скилл "делает всё" становится неуправляемым: баг непонятно где, улучшение одного кейса ломает другие. Решение — 3-5 фокус-скиллов, каждый отвечает за одну задачу, Claude координирует их сам (оркестрация). Баг находится мгновенно, улучшения не ломают соседние функции, не нужно строить заново.

## Итеративное накопление ("эффект 30 дней")
Цель Anthropic: «Claude на 30-й день работы должен быть заметно лучше, чем в первый». Механизм — не Claude "учится" сам, а пользователь/Claude вместе фиксируют находки в файл:
- После сессии — один вопрос: «разовая правка или в скилл навсегда?»
- Просить Claude самого предложить правки в `SKILL.md`/`tools/` на основе только что решённой задачи
- **Раздел "Gotchas" (типичные ошибки)** — самый ценный контент скилла на практике; не придумывается заранее, а дополняется по мере реальных факапов
- Органический рост: сначала личная sandbox-версия скилла, публикация в общий marketplace — только когда доказал полезность

## Тестирование и итерация
1. Начинать с оценки: гонять на representative задачах, искать конкретные пробелы
2. 2-3 реалистичных тест-промпта → `evals.json`
3. Сравнение with/without skill (baseline обязателен, иначе нельзя оценить, помогает ли скилл вообще)
4. Мониторинг реального использования, self-reflection при ошибках
5. Не переобучаться на 2-3 тест-кейса — обобщать, объяснять "почему", а не только "что делать"

## Частые ошибки
| Ошибка | Последствие |
|---|---|
| Обобщённый description | Скилл не триггерится (undertriggering) |
| SKILL.md > 500 строк без иерархии | Перегрузка контекста при каждой активации |
| Хардкод примеров вместо паттернов | Скилл не обобщается на новые случаи |
| MUST/NEVER без объяснения "почему" | LLM игнорирует правило без понимания причины |
| Нет bundled script для повторяющейся работы | Каждый вызов заново "изобретает колесо" токенами |
| Нет baseline-сравнения при тестировании | Невозможно оценить, действительно ли скилл помогает |

## Где хранятся и как приоритизируются (Claude Code)
Enterprise > Personal (`~/.claude/skills/`) > Project (`.claude/skills/`) > Plugin (namespace `plugin:skill`) — при совпадении имён старший уровень перекрывает младший. Custom commands слиты со skills: `.claude/commands/deploy.md` эквивалентен `.claude/skills/deploy/SKILL.md`. В монорепо скиллы из `packages/frontend/.claude/skills/` подхватываются при работе с файлами в этой директории.

## Токен-эффективность (замер из [[hook-4-pravila-claude-skills]])
Одна и та же задача (ответ на письмо в стиле пользователя): промт — 2700 токенов / 70 сек / стиль "плавает"; скилл `/draft-email` — 900 токенов / 25 сек / 100% попадание в стиль с первого раза. ×3 меньше токенов, ×2.8 быстрее.

## Универсальность принципа
Работает везде, где есть кастомная "обёртка" над моделью, не только в Claude Code: ChatGPT (Custom GPT + uploaded files как tools), Gemini (Gems), Cursor/Windsurf (`.cursorrules` + agents + скрипты в проекте), n8n/Make (workflow-шаблоны = композиция узлов), Vercel AI SDK (tool calling + system prompts в коде). Общий принцип: не начинать каждую сессию с нуля.
