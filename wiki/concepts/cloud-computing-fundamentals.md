# Основы облачных вычислений

## Определение
Базовая терминология и архитектура облачных вычислений — не про ИИ напрямую, загружено для более осознанного проектирования инфраструктуры вайбкодинг-проектов (см. обоснование темы в [[jarvis-personal-wiki]]). Источник — [[prostodevops-cloud-infrastructure]].

## Определение облака (NIST, 5 признаков)
Самообслуживание по требованию, доступ по сети откуда угодно, пул ресурсов/мультиарендность (одно физическое железо обслуживает много логически изолированных клиентов), эластичность (мощности растут/сжимаются под нагрузку), измеряемая оплата за фактическое потребление. "Чужой компьютер" ≠ облако — это просто хостинг без этих пяти свойств.

## Модели обслуживания (по уровню ответственности провайдера, снизу вверх)
- **Bare metal / своё железо** — полный контроль и полная эксплуатация на стороне пользователя.
- **IaaS** (Infrastructure as a Service) — провайдер даёт железо+виртуализацию (VM, сеть, диски); ОС/рантайм/приложение — забота пользователя. Пример: AWS EC2, Yandex Compute.
- **PaaS** (Platform as a Service) — провайдер держит ещё и ОС+рантайм; пользователю остаётся только код и его конфигурация. Пример: managed БД.
- **FaaS / Serverless** (Function as a Service) — провайдер запускает отдельные функции по событию (HTTP-запрос, сообщение в очереди, загрузка файла); оплата только за фактическое время выполнения, ноль — если событий нет. Пример: AWS Lambda, Yandex Cloud Functions. Компромисс — почти полная потеря контроля над средой выполнения и более сильный vendor lock-in.
- **SaaS** (Software as a Service) — готовый продукт целиком на стороне провайдера, пользователь работает только со своими данными внутри чужого приложения. Пример: Gmail.

## Модели размещения
Публичное (общий провайдер/железо — дёшево и быстро, но железо не ваше) / приватное (свой ЦОД, для чувствительных данных или по требованию закона) / гибридное (чувствительное и стабильное — у себя, пики нагрузки и эксперименты — в публичном облаке; самый частый выбор зрелых компаний) / мультиоблако (несколько публичных провайдеров ради отказоустойчивости и снижения зависимости от одного вендора).

## Механика под капотом
- **Гипервизор** режет физический сервер на изолированные VM (Xen у раннего AWS EC2 → KVM сегодня, на нём почти все российские облака; AWS Nitro выносит виртуализацию на отдельные железные карты, почти убирая оверхед).
- **Контейнеры** — изолированные процессы поверх общего ядра ОС, легче и быстрее VM; при масштабе ими управляет **Kubernetes** (планирование, перезапуск упавшего, масштабирование).
- **VPC** — программируемая виртуальная сеть поверх общей физической (файерволлы/маршруты как конфиг).
- **Хранилище**, три типа: блочное (быстрый диск для одной машины), объектное/S3-подобное (масштабируется практически бесконечно, дёшево — для бэкапов/медиа/статики), сетевая папка (общий доступ с многих машин).
- **Регион** — набор дата-центров в одной географической точке; **зона доступности** — физически отдельный ЦОД со своим питанием/сетью внутри региона, для отказоустойчивости при отказе одного ЦОДа целиком.

## Плюсы / минусы
Плюсы: capex→opex, скорость разворачивания, эластичность оплаты по факту нагрузки, managed-сервисы снимают рутину эксплуатации, глобальность в несколько кликов.
Минусы: vendor lock-in (тем глубже, чем больше проприетарных сервисов использовано), при стабильно высокой нагрузке 24/7 своё железо может оказаться дешевле в моменте (реальный тренд возврата части нагрузок на bare metal), исходящий трафик обычно платный и часто недооценённая статья расходов.

## Специфика РФ
152-ФЗ требует хранить/обрабатывать персональные данные россиян физически на территории РФ (с 2015), с 2023 — уведомление Роскомнадзора перед трансграничной передачей. После ухода AWS/Azure/GCP в 2022 рынок вырос вокруг Cloud.ru/Yandex Cloud/Selectel. Для критической инфраструктуры и госсектора вместо Ubuntu/RHEL используются сертифицированные отечественные ОС — лидер Astra Linux Special Edition (сертификация ФСТЭК), рядом RedOS и Alt Linux.
