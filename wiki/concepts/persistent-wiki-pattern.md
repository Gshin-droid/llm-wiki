# Persistent Wiki Pattern (вики вместо RAG)

## Определение
Паттерн построения персональной базы знаний, в котором LLM не просто извлекает релевантные фрагменты из сырых документов при каждом запросе (классический RAG), а постепенно строит и поддерживает персистентную, взаимосвязанную коллекцию markdown-файлов между пользователем и источниками. Знания компилируются один раз и поддерживаются в актуальном состоянии, а не собираются заново при каждом вопросе.

## Три слоя архитектуры
1. **Raw sources** — неизменяемая курированная коллекция исходников (источник истины)
2. **Wiki** — сгенерированные LLM markdown-файлы: саммари, страницы сущностей, концептов, синтеза
3. **Schema** ([[CLAUDE.md]]) — файл правил, описывающий структуру вики, конвенции и рабочие процессы

## Чем отличается от обычного RAG
| RAG (NotebookLM, ChatGPT files) | Persistent Wiki |
|---|---|
| Извлечение фрагментов при каждом запросе | Знания накапливаются один раз |
| Нет накопления между запросами | Перекрёстные ссылки и синтез уже проставлены |
| Противоречия не отслеживаются | Противоречия явно помечаются |
| LLM "открывает знания с нуля" | Вики становится богаче с каждым источником |

## Ограничения (уточнено источником mastodont-claude-obsidian-video)
Паттерн отлично работает на малых объёмах, но деградирует на масштабе:
- `index.md`/`CLAUDE.md` растут линейно с числом файлов → налог на контекст при каждом запросе (перечитывание всего индекса)
- Нет семантического поиска — только по ключевым словам/темам, не по смыслу
- Не рассчитан на большие объёмы данных

**Уточнение масштаба из первоисточника** ([[karpathy-llm-wiki-gist]]): index.md-подход хорошо работает примерно до ~100 источников и нескольких сотен страниц без необходимости embedding-инфраструктуры. За этим порогом — опциональный локальный поисковый движок **qmd** (`github.com/tobi/qmd`, гибридный BM25/vector + LLM re-ranking, CLI + MCP), а не обязательно [[hybrid-memory-architecture|переход на Pinecone/NotebookLM]]. Сейчас у этой вики 14 источников — до порога далеко.

## Формы ответа на Query — не только markdown-страница
Из первоисточника: ответ на вопрос может быть сравнительной таблицей, слайд-деком (Marp), графиком (matplotlib), canvas — не обязательно markdown-текстом. Пока в этой вики все `wiki/synthesis/`-страницы были обычным markdown — стоит иметь в виду другие форматы для будущих запросов.

## Другие практические детали из первоисточника
- Изображения из источников можно скачивать локально: Obsidian Settings → Files and links → Attachment folder path на `raw/assets/`, плюс горячая клавиша на "Download attachments for current file" — у нас `raw/assets/` создана, но не настроена (см. открытый пробел в [[obsidian]])
- Вики — это просто git-репозиторий markdown-файлов: версии, ветки, совместная работа достаются бесплатно (этот проект пока не под git)
- Паттерн применим не только к ИИ-теме: personal (здоровье/психология), research, чтение книги (аналог фан-вики вроде Tolkien Gateway), business/team, competitive analysis, due diligence
- Концептуальная связь с Vannevar Bush Memex (1945) — курируемое персональное хранилище с ассоциативными связями; недостающую часть его идеи (кто делает поддержку) теперь решает LLM

## Почему это работает
Утомительная часть баз знаний — не чтение, а "бухгалтерия": обновление ссылок, сводок, пометка противоречий, консистентность на десятках страниц. Люди забрасывают вики, потому что поддержка растёт быстрее ценности. LLM не устаёт и не забывает обновить ссылку — стоимость поддержки близка к нулю.
