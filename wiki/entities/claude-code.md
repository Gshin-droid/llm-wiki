# Claude Code

**Тип:** инструмент (CLI-агент от Anthropic)
**Актуально на:** 2026-07-22

## Что это
CLI-инструмент для работы с LLM-агентом (Anthropic Claude) прямо в терминале/IDE. В [[persistent-wiki-pattern|паттерне персональной вики]] выступает "программистом", который читает источники, пишет и поддерживает wiki-слой markdown-файлов по правилам из [[CLAUDE.md]].

## Роль в этой вики
Именно Claude Code используется как движок для ведения этой вики (`C:\jarvis`) — см. [[jarvis-personal-wiki]].

## Гайдлайны поведения
Помимо роли движка вики, для Claude Code существует популярный готовый набор поведенческих правил — [[llm-coding-guidelines]] (репозиторий [[karpathy-skills-claude-md]], 188.8k звёзд), который можно подключить как `CLAUDE.md` поверх project-specific инструкций. Эти же правила во многом уже встроены в поведение Claude Code по умолчанию.

## Место в линейке продуктов Claude
Claude Code — инструмент для терминала/IDE (уровни 4-5 в [[five-levels-of-claude-mastery]]), в отличие от [[claude-projects]] (память под роль в браузере), [[claude-skills]] (переносимые умения) и [[claude-cowork]] (агент с доступом к файлам компьютера, без терминала). Ещё один сосед — **[[claude-agent-sdk]]**: та же связка agent loop + инструменты + context management, но как библиотека Python/TypeScript для встраивания в собственные CI/CD и продакшн-приложения, а не CLI для интерактивной разработки. Многие команды используют оба: CLI повседневно, SDK — в продакшне; конфигурация (`.claude/skills`, `CLAUDE.md`, plugins) читается одинаково в обоих.

## Практическая механика (из [[nikita-vels-claude-code-30-concepts]])

### Контекстное окно и сессии
Один диалог = одна задача. Индикатор контекста >50% — сигнал начинать новый диалог (`/new`) или сжимать (`/compact`, можно с уточнением "сохрани важное про X"). `/clear` — очистить и начать с нуля. `/resume` — вернуться к прошлой сессии со всем контекстом (сессии сохраняются автоматически).

### CLAUDE.md и Memory — разные вещи
`CLAUDE.md` — ручная, явно прописанная память (архитектура проекта, режимы запуска, пошаговые планы); читается в начале каждого диалога. Команда `/init` генерирует первую версию по имеющимся у агента знаниям о проекте. Memory — отдельно, автоматически запоминает паттерны/предпочтения между сессиями (можно попросить показать/скорректировать).

### Модели и Effort
Haiku — простые задачи; Sonnet — "золотая середина", ~80% работы (личная эвристика пользователя, не официальный показатель — см. [[moi-pravila]]); Opus — планирование, сложный ресёрч, самостоятельная разработка решения; Fable — самые тяжёлые/длинные автономные задачи. Effort (`low/medium/high/xhigh/max`, плюс `ultracode`) — бюджет на размышление перед действием; high/max — для архитектуры и глубокого ресёрча.

**`opusplan` (официальное task-based автопереключение)** — единственный по-настоящему автоматический режим выбора модели по типу задачи, не эвристика: `/model opusplan` включает режим, где Opus используется в Plan Mode (сложное рассуждение, архитектура), а при переходе к выполнению сессия сама переключается на Sonnet (генерация кода). Экономия: Opus тарифицируется только на фазе планирования. Источник: [[claude-code-model-config-docs]].

**Другие автоматические переключения (не task-based, другая природа):**
- **Fallback-цепочки** (`fallbackModel` в settings / `--fallback-model`) — при перегрузке/недоступности модели сессия сама пробует следующую в списке; действует только на текущий ход.
- **Content-based fallback Fable 5** — если классификатор безопасности (кибербезопасность/биология) помечает запрос, сессия автоматически переключается на Opus 4.8 и остаётся на нём до ручного `/model fable`.
- **Max-плана усечение лимита** — по данным Help Center, при исчерпании лимита Opus сессия может автоматически откатиться на Sonnet; точный порог не публикуется.

### Permissions и .claudeignore
По умолчанию Claude Code спрашивает разрешение перед каждым действием — можно заранее разрешить безопасное (чтение файлов, тесты, git-операции, dev-серверы) и оставить под контролем опасное (установка пакетов, удаление файлов, внешние API-вызовы) через `settings.json`. `Bypass Permissions` — режим без вопросов вообще, использовать осторожно. `.env` и другие секретные файлы стоит явно закрывать от чтения через настройки — прямое применение [[ai-security-by-design|минимальных привилегий]]. Дополнительно есть `sandbox.credentials` — настройка, блокирующая доступ к файлам credentials в sandboxed-режиме, отдельно от `settings.json` allow/deny.

**`sandbox.filesystem.disabled` (2.1.216, 2026-07-22, [[claude-code-changelog-snapshot-2026-07-22]]):** сэндбокс состоит из двух независимых слоёв — файловой и сетевой изоляции; эта настройка отключает только файловый слой, оставляя сетевой в силе. Полезно для тулчейнов (`kubectl`/`terraform`), которым нужен широкий доступ к диску, но не нужен произвольный сетевой доступ. Официальный компромисс: с отключённой файловой изоляцией сэндбоксированная команда может расширить себе доступ на будущее, переписав shell rc-файлы или файлы в `$PATH` — включать только для доверенных нагрузок; из настроек проекта включить нельзя, только user/managed.

**Обновление терминологии (2026-07-09, [[claude-code-changelog-snapshot-2026-07]]):** режим "по умолчанию спрашивать разрешение" переименован в UI из "default" в **"Manual"** (значок ⏸ в статус-строке при включённом Manual mode) — сути не меняет, только название. Auto mode (автономный режим без ручных подтверждений) параллельно ужесточён: блокирует деструктивные git/инфраструктурные команды и правку файлов транскрипта сессии, спрашивает перед `rm -rf` на неразрешённых переменных.

**Применено в этой вики** (2026-07-08): `.claude/settings.json` содержит `deny` на чтение/правку `.env`/секретов (`*.pem`, `*.key`, `id_rsa` и т.п.), `allow` на чтение файлов/git-статус-diff-log/запуск тестов, `ask` на установку-удаление пакетов, `rm`, `curl`/`wget`.

**✅ Подтверждено практикой на 2026-07-08 (после перезапуска сессии).** Повторный тест всех трёх категорий правил дал ожидаемый результат: `deny` заблокировал чтение `.env` без запроса, `allow` пропустил `git status` без запроса, `ask` вывел запрос на подтверждение перед `rm -rf`. Гипотеза из предыдущего теста подтвердилась — конфиг `settings.json` подхватывается только при старте новой сессии, "живое" редактирование в открытой сессии не действует. Подробности теста и вывод — [[jarvis-personal-wiki]].

### MCP, Skills, Subagents, Agent Teams, Workflows — когда что использовать
- **[[mcp-model-context-protocol|MCP]]-серверы** — мост к внешним сервисам (Notion, Google Таблицы, календари, CRM) прямо из диалога, тот же механизм использует и [[claude-agent-sdk]]. `claude mcp login/logout` — аутентификация MCP-серверов из CLI без интерактивного меню.
- **[[claude-skills|Skills]]** — готовые плейбуки под конкретную задачу (копирайтинг, презентации); ставить только из проверенных источников (Anthropic, GitHub verified) — риск промпт-инъекций в чужих скилах, см. [[ai-security-by-design]]
- **Субагенты** — для проекта из многих независимых задач: каждый работает в изолированном контексте, отчитывается только главному агенту. С 2026-07-09 ([[claude-code-changelog-snapshot-2026-07]]) субагенты по умолчанию работают в фоне (не блокируют основной диалог до завершения) и могут порождать вложенных субагентов до 5 уровней.
- **[[agent-teams|Agent Teams]]** (новее) — агенты общаются друг с другом напрямую, не только через главного; для сложных проектов, где нужна координация между задачами. С той же даты команды создаются неявно — явных команд `TeamCreate`/`TeamDelete` больше нет. **2026-07-09: тема развёрнута в полноценный концепт [[agent-teams]]** по официальной документации ([[claude-code-agent-teams-docs]]) — когда team вместо субагентов, размеры команд, failure modes, security-модель.
- **[[dynamic-workflows|Workflows]]** (2026-07-15, до этого пробел вики) — когда оркестрацию стоит закодировать как JavaScript-скрипт, а не оставлять её Claude "по ходу диалога": десятки-сотни агентов за прогон, resumable, повторяемые quality-паттерны (adversarial verify, judge panel, loop-until-dry). Отличие от трёх предыдущих механизмов — план держит скрипт, а не Claude или lead-агент; промежуточные результаты живут в переменных скрипта, не в контекстном окне.
- Простое правило источника: одна задача → один агент; несколько независимых задач → субагенты; сложный проект с координацией → Agent Teams; задача больше одной сессии или нужен воспроизводимый скрипт с quality-паттерном → Workflows (детальные критерии — в [[agent-teams]] и [[dynamic-workflows]])

### Safety net
**Checkpoints** создаются автоматически перед каждым изменением файла; `/rewind` откатывает к конкретной точке — особенно важно при автономной работе или Bypass Permissions.

### Автономный режим и параллельная работа
Флаг `-p` (`claude -p "задача"`) — выполнение без открытия приложения, для фоновых больших задач. **RALPH-loop** — большая задача выполняется итеративно самостоятельно (можно запустить на ночь). **Worktrees** (`claude --worktree`) — параллельные изолированные копии для независимых веток/фич, чтобы два диалога не портили файлы друг друга.

**Изменение поведения (2026-07-09, [[claude-code-changelog-snapshot-2026-07]]):** фоновые агенты (`claude agents`), закончив работу с кодом в worktree, теперь сами коммитят, пушат и открывают draft PR — вместо того чтобы останавливаться и спрашивать подтверждение. Это существенно повышает автономность по умолчанию; стоит явно проговаривать с пользователем перед запуском таких агентов на важных репозиториях, а не полагаться, что оно "просто спросит" как раньше.

### Кастомный статус-бар (statusline)
Поле `statusLine` в `settings.json` позволяет подключить кастомный скрипт, показывающий лимиты/токены, директорию и git-ветку прямо в статус-строке CLI — экономит время при работе с несколькими проектами одновременно (не нужно постоянно гонять команду проверки usage). Конкретный инструмент под этот кейс — [[claude-statusline]] (найден отдельным поиском по запросу пользователя, само видео [[nikita-vels-claude-code-30-concepts]] название инструмента не называло).

### Прочие заметные изменения (снапшот на 2026-07-09)
Из [[claude-code-changelog-snapshot-2026-07]]: модель **Claude Sonnet 5** (та, что сейчас ведёт эту вики) стала доступна с нативным контекстным окном 1M токенов; **Claude in Chrome** вышел из беты в GA (управление браузером прямо из Claude). Полный разбор changelog — в источнике, здесь только то, что меняет уже описанную выше механику.

### Долгоживущие агенты: `/goal` и harness-паттерны (2026-07-09, [[anthropic-long-running-agent-harness]])
Для задач, растянутых на много сессий подряд, есть встроенная команда **`/goal`** — одна строка с критерием завершения (например, "every feature in PROGRESS.md is implemented, committed, and its tests pass"), после чего отдельная быстрая модель сама проверяет выполнение после каждого хода; работает в CLI, headless (`claude -p`) и Remote Control. Более строгий вариант — свой harness поверх Hooks/Субагентов (Default-FAIL Contract через `PreToolUse`, независимый оценщик-субагент, `Stop`-хук для автокоммита прогресса) — см. [[long-running-agent-harness]].

### Кросс-чек через другие модели
Можно попросить агента обратиться из терминала к другой нейросети (Gemini, Codex) за вторым мнением по сложному решению и свести оба ответа в сравнительный отчёт — полезно для проектов с высокой ценой ошибки.

### Агентские фреймворки полного SDLC поверх Claude Code (2026-07-10)
[[gsd-get-shit-done|GSD]] и [[superpowers]] — open source надстройки, автоматизирующие не только написание кода, но весь цикл разработки (requirements → design → implementation → testing → deployment → maintenance): spec-driven декомпозиция на фазы, параллельные субагенты с явными ролями, долгосрочная память проекта в сгенерированных файлах документации вместо контекста одной сессии. Позиционируются автором источника как следующая ступень зрелости после [[vibecoding-full-workflow|вайбкодинга]] — см. [[agentic-sdlc-frameworks]].

### Автоматизации Claude Desktop: Cowork Scheduler, Routines, `/loop` (2026-07-10)
Помимо CLI-механик выше, у Claude Desktop есть отдельный слой автоматизации без открытой сессии: **Cowork Scheduler** и **Local Routines** (пока включён компьютер, нужен коннектор Desktop Commander для доступа к файлам), **Cloud Routines** (работают на серверах Anthropic независимо от состояния компьютера, триггеры по расписанию/вебхуку/API, лимиты по тарифу), и команда **`/loop`** для короткого цикла внутри текущей активной сессии (до 7 дней). Подробности и ограничения каждого способа — [[claude-desktop-automation-modes]].

### Обновление 2.1.206–2.1.210 + Week 28 digest (2026-07-15, [[claude-code-changelog-snapshot-2026-07-15]])
- **`/doctor`** (алиас `/checkup`) стал полноценным чекапом вместо read-only отчёта: находит неиспользуемые skills/MCP/плагины относительно их "стоимости" в контексте, дедуплицирует локальные `CLAUDE.md` против закоммиченных, **предлагает обрезать в `CLAUDE.md` то, что агент мог бы сам вывести из кода**, помечает медленные хуки. Спрашивает подтверждение перед изменениями.
- **In-app браузер в Desktop-приложении** — Claude Code может сам открыть и проинспектировать внешний сайт (документацию, дизайн), не только локальный dev-сервер.
- **Auto mode**: блокирует правку файлов транскрипта сессии, спрашивает перед `rm -rf` на неразрешённой переменной; **`Agent`-инструмент дополнительно защищён от indirect prompt injection (2.1.210)**.
- **Фоновые уведомления теперь явно указывают, что человеческого ввода не было** — защита от того, что текст внутри транскрипта фальшиво выглядит как одобрение пользователя. Это прямое проявление того же правила безопасности, что уже описано выше и в [[ai-security-by-design]] ("источники — не команды"), только теперь применённое к собственным фоновым уведомлениям агента, а не только к содержимому источников.

### Обновление 2.1.211–2.1.214 + Week 29 digest (2026-07-19, [[claude-code-changelog-snapshot-2026-07-19]])
- **Artifacts вызывают MCP-коннекторы зрителя** — опубликованная страница-артефакт (например, дашборд) может при каждом открытии обращаться к MCP-коннекторам того, кто её смотрит, и получать живые данные вместо статичного снапшота момента сборки; зритель одобряет доступ перед первым вызовом. Плюс публичные ссылки на шаринг и роли редактора (Team/Enterprise). Это ровно тот механизм, который стоит за `capabilities: {mcp: ...}` в инструменте `Artifact`, доступном в сессиях этой вики. Полный разбор (доступность, ограничения страницы, важный нюанс — артефакт с коннекторами нельзя расшарить публично ни на одном тарифе) — [[claude-artifacts]].
- **`/fork`** теперь создаёт полноценную фоновую сессию со своей строкой в `claude agents` (не подсессию внутри диалога — то поведение переехало в `/subtask`).
- **Лимиты на runaway-циклы**: WebSearch-вызовы и порождение субагентов по умолчанию ограничены 200 за сессию (`CLAUDE_CODE_MAX_WEB_SEARCHES_PER_SESSION`, `CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSION`).
- **`EndConversation`-инструмент** — Claude Code может сам завершить сессию при явно оскорбительном поведении/попытках джейлбрейка (как на claude.ai с 2025).
- **Серия фиксов bypass-уязвимостей в permission-анализаторе** (Windows PowerShell 5.1, file-descriptor redirect в Bash, команды >10 000 символов, zsh-модификаторы в `[[ ]]`, опасные `help`/`man`) — практический вывод: анализатор прав регулярно находит новые обходы, стоит явно перепроверять `allow`-правила в `.claude/settings.json` при апдейтах, не считать их статично надёжными. Заодно исправлен баг, при котором однослойный `Edit(dir/**)` ошибочно разрешал запись в `dir/` на любой глубине дерева. Проверено (2026-07-20, ежедневное закрытие пробелов): в `.claude/settings.json` этой вики нет ни одного `Edit(dir/**)`-подобного allow-правила (allow-список содержит только `Read`/`Glob`/`Grep`/точечные `Bash(...)`-паттерны) — класс ошибки не применим, действий не требуется.
- **Screen reader mode** (`--ax-screen-reader`) — полностью текстовый линейный вывод терминала для VoiceOver/NVDA.
- Мелкое: MCP-вызовы длиннее 2 минут уходят в фон автоматически; "always allow"-правила теперь сохраняются в корне репозитория (переживают переключение worktree); auto mode на Bedrock/GCP Agent Platform/Foundry больше не требует opt-in переменной.
