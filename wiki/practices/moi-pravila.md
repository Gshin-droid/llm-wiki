# Мои правила — практики для меня как пользователя

**Живой документ** (слой `practices/`): правила, которые должен соблюдать я сам — их нельзя делегировать Claude. Синтез из всей вики 2026-07-09. Наглядная версия: `wiki/practices.html`. Правила для Claude Code — [[claude-code-practices]].

## А. Дисциплина работы с ИИ

Источник: [[seven-signs-of-old-junior]], [[five-levels-of-claude-mastery]].

1. **Ревьюить AI-код строже, чем код коллеги.** ИИ выдаёт уверенный, правдоподобный код — уверенность не равна правильности. Не мержить не глядя.
2. **Сначала думать самому, потом спрашивать ИИ.** Иначе атрофируется собственное умение решать — главный признак «старого джуна».
3. **Кросс-проверка на важных решениях:** архитектура, безопасность, деньги — спросить вторую модель (Gemini/Codex) и сравнить ответы.
4. **Доверие автономии наращивать постепенно** («пустая парковка»): сначала низкие ставки (черновики, ресёрч), потом код, потом фоновые агенты с auto-push.

## Б. Гигиена сессий Claude Code

Источник: [[claude-code-checklist-postoyannogo-ispolzovaniya]], [[nikita-vels-claude-code-30-concepts]].

5. **Один чат — одна задача.** Новая тема — новый диалог, не продолжение забитого контекста.
6. **Контекст >50% — реагировать:** `/compact` (если продолжаю ту же задачу, можно с «сохрани важное про X») или `/new` (если нет). *Эвристика из практики сообщества, не официальный порог.*
7. **Модель под задачу:** Sonnet — по умолчанию (~80% работы); Opus — планирование, архитектура, сложный ресёрч; effort high/max — только для глубокого анализа.
8. **Знать про Checkpoints/`/rewind` ДО включения Bypass Permissions**, не после аварии.
9. **Настроить permissions до начала проекта:** deny на `.env`/секреты, allow на безопасное, ask на опасное. Помнить: подхватывается при новой сессии.

## В. Решения, которые нельзя делегировать ИИ

Источник: практика рабочих вайбкодинг-проектов (детали — локально, не в этой вики), [[vibecoding-full-workflow]], [[10k-website-checklist]].

10. **Юрисдикция данных** (152-ФЗ vs GDPR) — вопрос юристу, не инженеру и не ИИ. Решить до выбора прод-хостинга.
11. **Бизнес-модель** (SaaS vs self-hosted, монетизация) — моё решение; ИИ может подготовить сравнение, но не выбрать.
12. **Деплой и установка пакетов — руками** (или с явным подтверждением каждый раз), не в автоматическом режиме агента.
13. **Вкус показывать скриншотами** (Dribbble/Awwwards), не словами — словесное описание дизайна ИИ интерпретирует шаблонно.
14. **Мой опыт легитимен:** если я реально работал со стеком (как с Postgres+nginx) — это достаточная причина отклонить «минимальную» рекомендацию ИИ. Фиксировать решение и причину в вики.

## Г. Привычки

Источник: [[skill-authoring-practical-rules]], [[long-running-agent-harness]], [[seven-signs-of-old-junior]], практика рабочих проектов.

15. **После фазы/фичи — verify-phase** (или попросить Claude; он напомнит сам — договорённость от 2026-07-09). Статические проверки не ловят всё: баг знака интервала и мёртвый guard нашлись только end-to-end и security-review.
16. **Конец сессии — один вопрос:** «разовая правка или в скилл навсегда?» 30 секунд — и следующая сессия умнее («эффект 30 дней»).
17. **TODO → тикет.** Временный фикс без тикета = вечный фикс.
18. **15-20% времени — на техдолг**, не 0%.
19. **Перед первым push нового репозитория** — глазами проверить, что секретов нет ни в одном коммите.
